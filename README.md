# compare_table

Скрипт `compare.py` собирает сравнительную таблицу изданий **Tantor DB** по нескольким мажорным версиям. Источник данных — пары файлов `conf.json` и `contrib.json` каждой версии. На выходе получаются два файла:

- `comparison.md` — таблица в Markdown (для Git/Gitea/GitHub и просмотра в редакторе);
- `comparison.html` — HTML-таблица для **вставки в Confluence** (Confluence корректно понимает `colspan`/`rowspan` при копи-пасте из браузера).

## Как устроена таблица

- **Столбцы** — пары `(версия, издание)` в том порядке, в котором они перечислены в `config.toml`. В HTML над изданиями есть «шапка второго уровня» с номером версии (через `colspan`).
- **Строки** идут тремя секциями подряд:
  1. `patches` — все патчи из `editions.<edition>.patches` (`conf.json`);
  2. `utils`   — все утилиты из ключей `editions.<edition>.utils` (`conf.json`);
  3. `contrib` — все имена расширений из `contrib[].name` (`contrib.json`), у которых текущее издание присутствует в `contrib[].editions`.
- **Содержимое ячеек** — символ `y`, если данный патч/утилита/расширение присутствует в этом издании; пусто — если нет.
- Внутри каждой секции строки **отсортированы по алфавиту** (без учёта регистра).

В `conf.json` имена изданий пишутся как `se1c` и `certified_2`, а в `contrib.json` — как `se-1c` и `certified-2`. Скрипт нормализует имена (отбрасывает `-` и `_`, приводит к нижнему регистру), поэтому `se1c == se-1c` и `certified_2 == certified-2` сравниваются корректно.

## Файл `config.toml`

В корне проекта лежит `config.toml`, который описывает, какие версии и издания попадают в таблицу:

```toml
[versions."18"]
editions = ["be", "se", "se1c"]
conf     = "/home/kot/repo/tantor-db-18_1/tantor/conf.json"
contrib  = "/home/kot/repo/tantor-db-18_1/tantor/contrib/contrib.json"

[versions."17"]
editions = ["be", "se", "se1c", "certified", "certified_2"]
conf     = "/home/kot/repo/tantor-db-17_9/tantor/conf.json"
contrib  = "/home/kot/repo/tantor-db-17_9/tantor/contrib/contrib.json"
# ...
```

Каждая секция `[versions."<номер>"]` обязана содержать:

| ключ       | тип            | смысл                                         |
|------------|----------------|-----------------------------------------------|
| `editions` | список строк   | имена изданий в нужном порядке столбцов       |
| `conf`     | строка (путь)  | абсолютный путь до `conf.json` этой версии    |
| `contrib`  | строка (путь)  | абсолютный путь до `contrib.json` этой версии |

Порядок секций в файле определяет порядок версий в таблице. Порядок изданий в `editions` определяет порядок столбцов внутри версии.

## Два источника JSON-данных

При запуске скрипт сразу спрашивает:

```
Использовать пути json из config.toml? [Y/n]:
```

- **Enter** или `yes` / `y` / `да` / `д` — берём `conf.json` и `contrib.json` по абсолютным путям из `config.toml`.
- **`no` / `n` / `нет` / `н`** — пути из `config.toml` **игнорируются**; для каждой версии скрипт читает файлы из папки рядом с собой:

  ```
  compare_table/
    compare.py
    config.toml
    14/conf.json
    14/contrib.json
    15/conf.json
    15/contrib.json
    ...
    18/conf.json
    18/contrib.json
  ```

  Имя папки должно совпадать с ключом версии в `config.toml` (`"18"` ↔ `18/`).

Любой другой ввод — переспрос. Если стандартный ввод закрыт (запуск из CI или из `cron`), используется значение по умолчанию (yes).

После выбора в stderr печатается одна из строк:

```
источник json: пути из config.toml
источник json: папки версий рядом со скриптом (/home/kot/repo-personal/compare_table)
```

## Валидация конфигурации

Перед обработкой `compare.py` строго проверяет конфигурацию и сразу прерывается, если что-то не так. Все обнаруженные проблемы выводятся **одним списком**, чтобы их можно было поправить за один проход:

```
error: config validation failed (3 problem(s)):
  - version '18': edition(s) ['ghost_edition'] not found in /path/to/conf.json (available: ['be', 'certified', 'certified_2', 'free', 'se', 'se1c'])
  - version '17': contrib.json not found: /path/that/does/not/exist.json
  - version '13': missing required key(s): editions, contrib
```

Что проверяется:

1. В `config.toml` есть хотя бы одна секция `[versions.*]`.
2. В каждой секции указаны все три обязательных ключа: `editions`, `conf`, `contrib`.
3. `editions` — непустой список строк.
4. Файлы `conf.json` и `contrib.json` существуют и являются валидным JSON-объектом.
5. Каждое издание из `editions` действительно присутствует в верхнеуровневом объекте `editions` соответствующего `conf.json` (если издание не найдено, в сообщении об ошибке выводится список реально доступных).

При ошибке скрипт завершается с кодом `2`. При успехе — `0`. При непредвиденном исключении — `1`, и в stderr выводится traceback.

## Запуск

### Из терминала

```bash
cd /home/kot/repo-personal/compare_table
./compare.py                    # дефолт: ./config.toml
./compare.py --config other.toml --md-out report.md --html-out report.html
```

CLI-аргументы:

- `--config PATH`    — путь до `config.toml` (по умолчанию `./config.toml`);
- `--md-out PATH`    — куда писать Markdown (по умолчанию `./comparison.md`);
- `--html-out PATH`  — куда писать HTML (по умолчанию `./comparison.html`).

Относительные пути в этих аргументах резолвятся **относительно директории скрипта**, потому что `compare.py` в начале своей работы делает `os.chdir` в свою папку (это нужно для запуска из файлового менеджера, где cwd — домашняя папка).

### Через файловый менеджер

У `compare.py` есть шебанг и установлен бит исполнения, поэтому в Nautilus / Nemo / Caja в контекстном меню по правой кнопке мыши доступен пункт **«Запустить как программу»**.

Для этого нужно:

1. Один раз в файловом менеджере включить запуск исполняемых текстовых файлов (в Nautilus: `Settings → Behavior → Executable Text Files → Run them`).
2. Кликнуть правой кнопкой по `compare.py` → **«Запустить как программу»** (или просто двойной клик при включённой настройке).

Терминал откроется, скрипт спросит про источник json, отработает и в конце дождётся `Enter` — чтобы окно не схлопнулось мгновенно и можно было прочитать вывод.

## Что лежит в репозитории и что нет

Файл `.gitignore` устроен «инверсно»: всё в директории игнорируется, кроме явно перечисленных:

```
/*
!compare.py
!config.toml
!README.md
```

То есть в git попадают только три этих файла. Папки версий (`14/`-`18/`) и сгенерированные `comparison.md` / `comparison.html` — локальные артефакты и в репозиторий не коммитятся.

## Требования

- **Python 3.11+** — используется stdlib-модуль `tomllib`.
- **Python 3.7–3.10** — тоже подходит, но нужно установить пакет [`tomli`](https://pypi.org/project/tomli/):

  ```bash
  pip install tomli
  ```

  Скрипт автоматически делает `try: import tomllib / except ModuleNotFoundError: import tomli as tomllib`.

Внешних зависимостей кроме `tomli` (только на старом Python) нет — всё на стандартной библиотеке.

## Коды выхода

| код | смысл                                                              |
|-----|--------------------------------------------------------------------|
| `0` | успешно, оба файла записаны                                        |
| `1` | непредвиденное исключение (в stderr напечатан traceback)           |
| `2` | проблема со входом: нет `config.toml`, битый TOML, ошибка валидации |
